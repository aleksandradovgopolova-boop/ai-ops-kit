# Урок: новый workflow-контракт без маршрута к нему невидим для системы

- **Источник:** живая демонстрация маршрутизации в сессии выпуска v2.1 (первый улов
  dogfooding); исправлено в v2.1.1.
- **Owner:** repository-memory-curator (зафиксировано при memory-capture починки).
- **Дата проверки:** 2026-07-09.
- **Условие устаревания:** появление кросс-реестрового валидатора достижимости
  workflow (кандидат в ROADMAP) делает урок автоматизированным правилом.

## Что случилось

В v1.3–v2.1 добавили контракты VISUAL/ANALYTICS/INSIGHTS/ADOPTION с
`selection_criteria.task_type`, но `workflow_rules` в routing-policy знали только
четыре MVP-имени, а fallback движка возвращал task_type как имя workflow — роутер
отдавал несуществующие маршруты (`workflow: "ui-change"`). Каждый реестр по
отдельности был валиден; расхождение жило между реестрами и не ловилось ни одним
валидатором.

## Урок

1. Определение workflow (selection_criteria) и маршрутизация к нему обязаны иметь
   один источник истины: с v2.1.1 движок читает selection_criteria из
   workflows.yaml напрямую, policy оставлены только эскалации.
2. Passthrough-fallback («верни вход как ответ») маскирует ошибки конфигурации —
   fallback должен быть явным дефолтом с причиной в reasons.
3. Чек-лист нового контракта: стадии + gates + selection_criteria + **прогон
   ai_route.py на его task_type'ах** (теперь закреплено selftest-сценариями).
