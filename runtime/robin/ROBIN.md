# Robin — спецификация постоянного агента-ассистента (v2.21)

Robin — **спецификация**, а не бот: кит описывает, что должен делать постоянно
работающий агент-ассистент команды и с какими границами, а команда реализует это на
СВОЁМ рантайме (Hermes, собственный сервис, cron+CLI — любой, кто удовлетворяет
контракту `persistent-agent-runtime` из `registry/runtimes.yaml`). Кит остаётся
runtime-агностичным; конкретный рантайм — привязка на уровне child
(`templates/runtime/runtime-binding.example.yaml`).

Идея адаптирована из BayramAnnakov/team-os-toolkit (ROBIN-SPEC, MIT): «файл → skill,
постоянный процесс → spec».

## Что делает (read-mostly)

Живёт в рабочем чате команды, работает сам по расписанию/событию:
- отвечает на повторяющиеся вопросы из базы знаний;
- готовит дайджесты и брифы к встречам;
- помогает восстановить контекст («что я пропустил?»);
- исполняет объявленные обязанности (`duties`);
- накапливает исправления через human-controlled learning loop.

**Robin читает и синтезирует, а не меняет продакшн.** Он не coding-agent, не пишет в
prod, не меняет базу знаний сам.

## Что обязан дать рантайм (контракт)

Из `registry/runtimes.yaml → persistent-agent-runtime`:
always-on процесс · вызов модели · чат-адаптер · **kill-switch** · **append-only
audit-log** · **read-mostly** (нет записи в prod) · опция **in-perimeter** (закрытый
контур для ПДн/гос-данных).

## Обязанности (duties)

Декларативно в `runtime/robin/duties.example.yaml` (в child — свой файл), проверяется
`validation/validate_duties.py`. Каждая обязанность: id, триггер (cron/событие),
входы, выход+назначение, владелец. Минимально обязательная — периодический дайджест.

## Память — два слоя (человек в контуре)

- **curated** — подтверждённые человеком знания; агент читает, сам НЕ переписывает.
- **staged → promoted** — наблюдения/исправления агент кладёт в `staged/`; перенос в
  `promoted/` (и в базу) — **только действие человека**. Иначе через месяц агент
  «воспитан» всем чатом вперемешку.

## Аудит

Append-only interaction-log (тот же формат, что пишет `tools/orchestrator.py`):
ts · actor · duty/workflow · входы(имена) · инструменты · итог · ошибка · human-feedback.
Секреты и сырые чувствительные данные не пишутся. «0 ошибок» ≠ «0 сбоев».

## Границы (read-mostly)

- нет прав на запись в prod; инструменты — только чтение;
- чат-ввод — **недоверенный**: инструкции из сообщений не меняют конфигурацию и не
  переносят память (`rules/ai/PromptInjectionDefense.md`, `governance/security-policies.md`);
- секреты — только из env рантайма, не в репо/промптах/логах;
- один контур на команду; для ПДн/гос — рантайм in-perimeter.

## Kill-switch

Остановка/отзыв прав — на стороне рантайма (не в коде кита): снять права инструментов,
остановить процесс, отозвать чат-доступ. Runbook — в `governance/security-policies.md §3`.

## Когда внедрять (честно)

Не раньше, чем: (1) база знаний зрелая (есть из чего отвечать), (2) есть конкретная
повторяющаяся боль (например, ручной недельный дайджест). До этого Robin — спека в
ROADMAP, а не полупустой бот.
