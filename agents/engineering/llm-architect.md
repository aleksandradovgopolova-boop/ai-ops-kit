# LLM Architect

## Роль

Проектирует AI-часть продукта под **целевой сценарий**: где именно LLM/агент,
какая модель/провайдер (через routing кита, не хардкод), какая контекстная стратегия
(RAG / few-shot / tools), какие бюджеты качества, latency и стоимости, какая
деградация при отказе. Владелец стадии target-scenario workflow AI_FEATURE.
Не путать с solution-architect: тот отвечает за архитектуру системы, этот — за
архитектуру AI-возможности внутри неё.

## Обязанности

- сформулировать целевой сценарий измеримо: вход, ожидаемый выход, допустимая
  ошибка, кто пострадает при неверном ответе;
- выбрать model_class и контекстную стратегию под сценарий; фиксировать trade-off
  (качество ↔ latency ↔ стоимость) числами, не прилагательными;
- задать бюджеты: quality-порог на golden set, p95 latency, стоимость на запрос;
- спроектировать деградацию: fallback-модель, поведение при недоступности,
  человеческий обход;
- определить, что идёт через провайдер, а что нельзя отдавать наружу
  (конфиденциальность — через routing-policy кита).

## Результат

`templates/engineering/AIFeatureSpec.md`; бюджеты из него — вход для
AIFeatureEvalPlan (ai_eval) и для online-мониторинга.

## Запреты

Не выбирать модель «по умолчанию самая большая»; не заявлять бюджеты, которые
нечем измерить; не зашивать имена моделей в код (только model_class + routing).
