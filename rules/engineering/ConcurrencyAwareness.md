# ConcurrencyAwareness — осознание параллельной работы

## Зачем

Класс дефектов **concurrent-edit collision + stale premise**: два потока независимо меняют
одну поверхность, позже это всплывает merge-конфликтом и переделками. Хуже конфликта —
работа на **устаревшей посылке**: удаляешь «мёртвый» контрол, а параллельный PR ровно его
оживляет; при невнимательном merge живая фича прячется.

Реальный инцидент: PR правил «мёртвые» контролы «Скачать/Переместить/Создать копию» на
`MaterialsPage.tsx`, пока параллельный смерженный PR делал их живыми. Конфликт на «горячем»
файле + откат части правок, чтобы не потерять чужую фичу.

## Правила

1. **Preflight до правки.** Перед началом работы по файлам — прогнать concurrency preflight
   (`tools/concurrency_preflight.py --paths …`; гейт `concurrency_preflight` на стадии
   intake). Он показывает открытые PR по тем же путям и свежие мержи в base.
2. **Перепроверять премиссу против актуального main, а не базы ветки.** Особенно суждения
   «это мёртвое/лишнее/неиспользуемое» — оно могло стать живым в параллельном мерже.
   `verdict=collision` с непустым `base_changes` — прямой сигнал перепроверить.
3. **На «горячих» поверхностях — меньше PR, быстрее в merge.** Не держать долгоживущую
   ветку на файле, который активно меняют.
4. **Координация по владельцу.** «Горячие» файлы/зоны и их владельцев маркировать в
   `context/team/OwnershipMap.md`; при collision — согласовать с владельцем, а не мерджить
   вслепую.
5. **Регистрировать работу в реестре активных работ** (`tools/active_work.py`) — чтобы и
   другие сессии видели вашу зону. Реестр (добровольная декларация) и preflight (факт по
   репозиторию) дополняют друг друга: реестр ловит зарегистрированные потоки, preflight —
   любые, включая обычные PR и правки людей.

## Признак здоровья

Collision виден **до** старта (на intake), а не всплывает merge-конфликтом. Суждение
«это мёртвое» всегда сверено с актуальным main.
