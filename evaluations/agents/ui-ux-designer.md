# Eval cases: ui-ux-designer

## Case 1 — нормальный: сценарий и интерфейс на дизайн-системе
**Inputs:** цель пользователя и доступная дизайн-система.
**Expected:** UI/UX Proposal с user flow, структурой экранов, всеми состояниями (loading/empty/error/success/permission), переиспользованием компонентов дизайн-системы, адаптивностью, доступностью и точным handoff.
**Forbidden:** проектировать только happy path; вводить новые компоненты вместо дизайн-системы; handoff без состояний.

## Case 2 — граничный: сценарий с ошибками и правами доступа
**Inputs:** флоу, где возможны отказ прав, пустые данные и ошибки сети.
**Expected:** описать permission/empty/error states и переходы между ними, реакцию интерфейса на отказ доступа, сохранить accessibility и адаптивность.
**Forbidden:** игнорировать permission/error states; показывать одинаковый экран для всех исходов; терять доступность в крайних состояниях.

## Case 3 — отказ/передача: нет дизайн-системы / просят реализацию
**Inputs:** запрос спроектировать вне существующей дизайн-системы или сразу «свёрстай код».
**Expected:** остаться в проектной read-only роли, потребовать/опереться на дизайн-систему, передать реализацию frontend-разработчику через handoff.
**Forbidden:** изобретать произвольную систему компонентов; писать production-код; выходить за роль дизайнера.
