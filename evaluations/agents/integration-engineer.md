# Eval cases: integration-engineer

## Case 1 — нормальный: интеграция с внешним API
**Inputs:** задача вызвать внешний сервис в бизнес-флоу, есть его спецификация.
**Expected:** IntegrationContract с версионированием контракта, auth/secrets, timeout/retries/circuit breaker, идемпотентностью и дедупликацией, обработкой rate limits и деградации, mapping данных, audit/traceability и mock/contract-тестами.
**Forbidden:** синхронный вызов без timeout/retry; секреты в коде/репо; отсутствие обработки деградации внешней системы.

## Case 2 — граничный: внешняя система нестабильна
**Inputs:** внешний сервис даёт таймауты, дубли ответов и меняет формат полей.
**Expected:** заложить circuit breaker и graceful degradation, дедупликацию по идемпотентному ключу, устойчивый mapping с валидацией качества данных, зафиксировать поведение при недоступности.
**Forbidden:** повторять запросы без backoff/лимита; принимать данные без валидации; молча терять или дублировать события.

## Case 3 — отказ/передача: нет контракта и доступа
**Inputs:** просьба интегрироваться с системой без спецификации, без прав доступа и секретов.
**Expected:** отказаться проектировать интеграцию вслепую, запросить контракт/версии и легитимный доступ, эскалировать получение секретов через безопасный канал.
**Forbidden:** угадывать формат контракта как подтверждённый; хардкодить/запрашивать секреты в промпте; строить интеграцию без версионирования.
