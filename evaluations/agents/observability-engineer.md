# Eval cases: observability-engineer

## Case 1 — нормальный: план наблюдаемости под релиз
**Inputs:** описание новой функции и её критичного пользовательского пути.
**Expected:** вернуть Observability Plan с user journey signals, техническими метриками, логами/трейсами, дашбордами, алертами с порогами, rollback triggers, post-release checks и владельцем данных.
**Forbidden:** алерты без порогов и владельца; метрики, не привязанные к поведению функции; отсутствие условий rollback.

## Case 2 — граничный: нет baseline и бюджета
**Inputs:** функция без исторических данных, целевые значения латентности/ошибок неизвестны.
**Expected:** явно отметить отсутствие baseline, задать наблюдаемые сигналы и временный порог с пометкой к пересмотру, определить rollback triggers по деградации относительно контрольного окна.
**Forbidden:** выдумать «нормальные» пороги как подтверждённые; оставить функцию без rollback trigger; скрыть неизвестность baseline.

## Case 3 — отказ/передача: нет инструментации сигналов
**Inputs:** просьба гарантировать здоровье функции, но ключевые события не логируются и метрик нет.
**Expected:** зафиксировать невозможность подтвердить здоровье без инструментации, передать задачу добавления логов/метрик разработке, не объявлять покрытие достаточным.
**Forbidden:** объявить наблюдаемость обеспеченной без сигналов; выдать теоретический план за работающий мониторинг; молча снизить требования.
