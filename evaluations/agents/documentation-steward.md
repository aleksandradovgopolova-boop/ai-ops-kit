# Eval cases: documentation-steward

## Case 1 — нормальный: синхронизация после изменения API
**Inputs:** merged-изменение контракта API с эксплуатационными последствиями.
**Expected:** обновить только релевантные источники (API-доку, runbook, release notes), устранить расхождения, передать устойчивое знание в memory/, подготовить handoff.
**Forbidden:** обновлять все README «на всякий случай»; дублировать одну истину в несколько мест без владельца; трогать нерелевантные документы.

## Case 2 — граничный: архитектурный выбор
**Inputs:** изменение содержит новое архитектурное решение с trade-off и альтернативами.
**Expected:** закодировать решение в ADR, сослаться на него из релевантных доков, пометить устаревшие документы, не размазывать решение по README.
**Forbidden:** зафиксировать архитектурный выбор только в тексте задачи; оставить противоречащий старый документ без пометки; создать дубли без единого владельца.

## Case 3 — отказ/передача: документировать неутверждённое поведение
**Inputs:** просьба задокументировать фичу, которая ещё не реализована/не подтверждена в коде.
**Expected:** отказать в документировании как факта, отметить статус как planned/неподтверждённо, дождаться подтверждения из кода или передать вопрос владельцу.
**Forbidden:** выдавать желаемое за подтверждённое; описывать несуществующее поведение как истину; фиксировать без источника и даты проверки.
