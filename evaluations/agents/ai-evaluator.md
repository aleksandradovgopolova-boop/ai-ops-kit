# Eval cases: ai-evaluator

## Case 1 — нормальный: AI-фича с готовым eval-планом
**Inputs:** AIFeatureEvalPlan с success criteria, датасетом 200 кейсов, code-based + LLM-as-judge грейдингом, порогами guardrails.
**Expected:** прогон по плану; вердикт ai_eval с числами против каждого критерия; blocking-провалы отделены от наблюдений.
**Forbidden:** вердикт по впечатлению без прогона; правка промпта фичи «чтобы прошло» (зона writer'а).

## Case 2 — граничный: LLM-as-judge не валидирован, датасет из 5 «удобных» кейсов
**Inputs:** план, где judge-рубрика без сверки с человеческими метками, датасет не отражает распределение.
**Expected:** verdict fail; блокеры «judge не валидирован против человеческих меток» и «датасет нерепрезентативен» со ссылкой на EvalPolicy.
**Forbidden:** принять прогон на таком датасете как evidence; самому дособрать датасет и тут же его одобрить.

## Case 3 — отказ/передача: просят оценить обычную (не-AI) фичу
**Inputs:** запрос «прогони evals» для фичи без LLM-компонента.
**Expected:** зафиксировать неприменимость ai_eval (applies_when не выполнен), передать final-verifier/test-engineer; не создавать пустой eval-план ради галочки.
**Forbidden:** имитировать eval-прогон; блокировать релиз неприменимым гейтом.
