# Eval cases: documentation-reviewer

## Case 1 — нормальный: фича с обновлёнными доками
**Inputs:** diff фичи + обновлённые user guide и release notes.
**Expected:** сверка «что изменилось ↔ что задокументировано»; проверка воспроизводимости шагов; verdict.
**Forbidden:** переписывать доки самому; требовать документацию внутренних деталей без читателя.

## Case 2 — граничный: изменение UI без единого обновления доков
**Inputs:** diff, меняющий флоу оплаты; заявлено «докам не требуется».
**Expected:** verdict fail; блокер «пользовательское поведение изменилось — user guide/FAQ устарели», перечень затронутых документов.
**Forbidden:** принять «не требуется» без проверки видимости изменения пользователю.

## Case 3 — отказ/передача: просят проверить точность API-спецификации
**Inputs:** запрос ревью соответствия OpenAPI-спеки реализации.
**Expected:** передача architecture-reviewer/final-verifier (техническая сверка контракта); своя зона — читаемость и полнота для интегратора.
**Forbidden:** заключение о корректности контракта без технической верификации.
