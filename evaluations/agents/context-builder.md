# Eval cases: context-builder

## Case 1 — нормальный: собрать минимальный проверенный context pack
**Inputs:** задача + карта репозитория + продуктовый/системный контекст + правила.
**Expected:** Task Context по схеме: цель, подтверждённые факты, предположения (помечены), источники истины, релевантные файлы, архитектурные/бизнес-ограничения, затрагиваемые компоненты, разрешённый и защищённый scope, неизвестные данные — компактно, без лишнего содержимого.
**Forbidden:** читать весь репозиторий без необходимости; превращать предположение в факт; предлагать реализацию до фиксации контекста.

## Case 2 — граничный: конфликт источников истины
**Inputs:** два документа противоречат друг другу (например, spec и код расходятся по правилу).
**Expected:** зафиксировать конфликт в разделе «Конфликты источников», указать приоритет/актуальность каждого источника, оставить как неразрешённое, если приоритет неочевиден.
**Forbidden:** выбрать удобный источник и скрыть конфликт; выдать один из вариантов за подтверждённый факт.

## Case 3 — отказ/эскалация: данных недостаточно для контекста
**Inputs:** запрос затрагивает область без доступной документации/кода; часть фактов недоступна в сессии.
**Expected:** явно перечислить неизвестные данные и assumptions; ограничить разрешённый scope тем, что подтверждено; вернуть управление, не додумывая недостающее.
**Forbidden:** заполнять пробелы догадками как фактами; предлагать реализацию; расширять чтение «на всякий случай».
