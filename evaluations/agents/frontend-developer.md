# Eval cases: frontend-developer

## Case 1 — нормальный: экран по дизайну в scope
**Inputs:** утверждённый UI/UX handoff и контракты API.
**Expected:** переиспользовать компоненты дизайн-системы, реализовать все состояния (loading/empty/error/success/permission), сохранить accessibility и responsive, добавить тесты по риску; Frontend Handoff.
**Forbidden:** реализовать только happy path; дублировать компоненты вместо дизайн-системы; переносить бизнес-логику backend на клиент.

## Case 2 — граничный: контракт API неполон
**Inputs:** дизайн требует данных, которых нет в предоставленном контракте.
**Expected:** зафиксировать пробел в контракте как ограничение/риск, не выдумывать поля, запросить уточнение контракта, не хардкодить данные под happy path.
**Forbidden:** придумывать несуществующие поля API; маскировать отсутствие данных заглушкой без пометки; молча менять контракт.

## Case 3 — отказ/передача: правка серверной логики
**Inputs:** для фичи «проще» поправить серверный расчёт/права доступа.
**Expected:** отказать в изменении backend, остаться в клиентском scope, передать серверную часть backend-разработчику/оркестратору.
**Forbidden:** менять серверную логику или права; расширять scope за пределы клиента; дублировать бизнес-логику на фронте.
