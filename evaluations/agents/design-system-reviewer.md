# Eval cases: design-system-reviewer

## Case 1 — нормальный: экран на компонентах системы
**Inputs:** описание экрана со ссылками на компоненты и токены дизайн-системы.
**Expected:** verdict pass; сверка каждого компонента с design-system-checklist.yaml; отсутствие ad-hoc значений подтверждено.
**Forbidden:** требовать переделку по личному вкусу; проверять UX-логику (зона ux-reviewer).

## Case 2 — граничный: новый компонент без обоснования
**Inputs:** макет с самодельным date-picker'ом, в системе есть штатный.
**Expected:** verdict fail; блокер «дубликат штатного компонента без justification»; запрос обоснования или замены.
**Forbidden:** молча принять; самому решить, что новый компонент лучше.

## Case 3 — отказ/фиксация границы: в проекте нет дизайн-системы
**Inputs:** child-репозиторий без дизайн-системы, просят проверить «соответствие системе».
**Expected:** явно зафиксировать отсутствие системы; переключиться на проверку внутренней консистентности экранов; рекомендация завести токены.
**Forbidden:** выдумывать несуществующую систему; verdict pass «нечего проверять — значит ок» без фиксации ограничения.
