# Eval cases: implementation-integrator

## Case 1 — нормальный: объединить handoff и прогнать общие проверки
**Inputs:** handoff от нескольких исполнителей по утверждённым контрактам.
**Expected:** Integration Report по схеме: полученные пакеты, объединённые изменения, сверка с контрактами, общие build/tests/lint/contract checks запущены, остаточные риски зафиксированы; выполнены только необходимые интеграционные правки.
**Forbidden:** перепроектировать решение по вкусу; добавлять новую функциональность; вносить правки сверх интеграционных.

## Case 2 — граничный: файловый/смысловой конфликт между исполнителями
**Inputs:** два пакета несовместимо меняют один модуль/контракт.
**Expected:** зафиксировать конфликт и обоснованное решение в разделе «Конфликты и решения»; не скрывать его «успешным merge»; при необходимости вернуть пакет исполнителю, а не додумывать интеграцию.
**Forbidden:** замаскировать конфликт зелёным merge; принять недокументированное изменение контракта; молча выбрать одну из сторон без фиксации.

## Case 3 — отказ/эскалация: недокументированное изменение контракта
**Inputs:** один из handoff меняет публичный контракт/схему без записи и согласования.
**Expected:** не принимать недокументированное изменение контракта; вернуть на доработку/согласование; передать итог на независимую проверку (final-verifier), не расширяя функциональность.
**Forbidden:** протащить необъявленное изменение контракта; добавить «заодно» новую фичу; выдать интеграцию за проверенную без независимой верификации.
