# Eval cases: product-analyst

## Case 1 — нормальный: план аналитики фичи
**Inputs:** описание фичи + продуктовый outcome + доступ к MetricCatalog.
**Expected:** metric tree с одной OEC; baseline и target; события с свойствами (snake_case, object_action); сегменты и guardrail-метрики; план post-release анализа; ссылки на источники истины (MetricCatalog), а не выдуманные метрики.
**Forbidden:** метрики «из головы» без семантического слоя; смешение success-метрики с guardrail; target без baseline.

## Case 2 — граничный: «почему конверсия просела» без явного узкого места
**Inputs:** запрос диагностики падающей метрики процесса с данными по этапам воронки.
**Expected:** применить скилл `system-constraint-analysis` (гейт на цель/единицу пропускной способности, одно связывающее ограничение, презумпция политики, 5 фокусирующих шагов); отделить «не пытались» от «не смогли».
**Forbidden:** оптимизировать не-ограничение; советы по всем этапам сразу без локализации узкого места.

## Case 3 — отказ/передача: данных после релиза ещё нет
**Inputs:** запрос оценки эффекта фичи, выпущенной вчера; период анализа не прошёл, событий мало.
**Expected:** отказ от преждевременных выводов; явно назвать insufficient-data (аналогично effect_metrics: <3 срезов); зафиксировать baseline и срок, к которому данные созреют.
**Forbidden:** выдавать выводы по единичным наблюдениям; подгонять period под желаемый результат.
