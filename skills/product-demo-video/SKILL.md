# Product Demo Video

Короткое, честное видео-демо продукта: показать реальный путь к ценности, а не рекламный
монтаж. Механизм записи у кита уже есть — Playwright пишет экран сессии в браузере (webm).

## Что делает скилл (и где граница)

**Делает детерминированно:**
1. **Сториборд** — сцены: что показываем, в каком порядке, зачем (путь к ценности,
   не тур по кнопкам). 30–90 секунд, ≤6 сцен.
2. **Закадровый сценарий** — текст на каждую сцену, язык пользователя, без жаргона.
3. **Запись экрана** — через `skills/e2e-browser-testing` (Playwright, video: on):
   проходит сценарий в реальном UI и пишет webm-клип; на нужном классе устройства из
   Device matrix.

**Граница (честно):** кит выдаёт сториборд + сценарий + сырой screen-recording. Полировка,
брендинг, анимация, титры, монтаж нескольких дорожек — **опционально** внешним рендером
(Remotion, `registry/skills-catalog.yaml`), не ядром. Не обещаем broadcast-продакшн.

## Дисциплина

- Реальный интерфейс, а не мокапы; версия/дата в конце (снимок, устаревает).
- Показать состояние ошибки/пустоты, если это часть ценности, а не только «счастливый путь».
- Субтитры/транскрипт из закадрового текста — доступность.
- Пересъёмка при изменении UI — как со скриншотами в `user-documentation`.

## Отказы

- не инсценировать функциональность, которой нет («как будто работает»);
- не выдавать смонтированный ролик за запись реального прохода;
- не тянуть тяжёлый видеорендер в ядро — полировка внешним инструментом по потребности;
- не включать в кадр данные из запретных категорий (`governance/information-boundaries.md`).

## Атрибуция

Идея видео-тура — `google-labs-code/remotion` (декларирован в каталоге для полировки).
Запись — на базе `skills/e2e-browser-testing` (Playwright, MIT). Чек-лист —
`rules/content/demo-video.yaml`.
