# System Constraint Analysis

Пропускная способность системы равна пропускной способности её ограничения. Улучшишь
что-либо ещё — не улучшишь ничего. Задача скилла — найти **одно** связывающее
ограничение и дать действия в правильном порядке, а не раздать общие советы по
оптимизации.

Инструмент-агностично: на вход — описание процесса (текст, файл, интервью), на выход —
markdown. Ничего не сочиняет за человека: если шага/ёмкости в описании нет, спрашивает,
а не выдумывает.

## Фаза 0 — гейт на цель

Нет единицы пропускной способности и цели (сколько чего к какому сроку) — нет
диагностики. Сначала цель, потом ограничение.

## Фаза 1 — извлечь поток

Шаги процесса, ёмкости, спрос, где копится работа (очереди), какие правила управляют
каждым шагом, по каким метрикам его гоняют. Дословно из описания; недостающее —
спросить.

## Фаза 1.5 — классифицировать систему

Потоковая (стабильное узкое место) / ограниченная спросом / исследовательская.
**Отказ навязывать ограничение** там, где его нет: у demand-constrained системы узкое
место — спрос, а не внутренний шаг; исследовательской нужна не оптимизация потока.

## Фаза 2 — локализовать ограничение

Два сходящихся пути: (1) иди вверх по потоку за кучей накопленной работы; (2) найди шаг
с потолком пропускной способности. **Презумпция: это политика (правило), а не ресурс** —
там, где смена правила подняла бы ёмкость, ограничение называем политикой, а не «не
хватает людей/машин».

## Фаза 3 — 5 фокусирующих шагов

В порядке, где бесплатное раньше платного:

1. **Identify** — ограничение названо (фазы выше).
2. **Exploit** — выжать максимум из ограничения как есть (бесплатно).
3. **Subordinate** — подчинить всё остальное ритму ограничения (бесплатно).
4. **Elevate** — расширить ограничение (платно) — только после Exploit/Subordinate.
5. **Repeat** — ограничение сдвинулось; вернуться к шагу 1.

Каждый шаг судится по T↑ / I↓ / OE↓ (throughput вверх / inventory вниз / operating
expense вниз).

## Фаза 4 — предупреждения

Назвать ловушку локальной эффективности (оптимизация не-ограничения создаёт видимость
работы) и **кто конкретно будет сопротивляться** subordinate (подчинение своего участка
чужому ритму бьёт по локальным KPI).

## Отказы (не обсуждаются)

- не оптимизировать не-ограничения;
- не рекомендовать Elevate раньше Exploit/Subordinate;
- не начинать без заявленной цели/единицы пропускной способности;
- не навязывать единственное ограничение demand-constrained системе;
- не называть ограничение-политику ресурсным ограничением;
- не выдумывать отсутствующие шаги/ёмкости;
- не писать код и не менять файлы.
