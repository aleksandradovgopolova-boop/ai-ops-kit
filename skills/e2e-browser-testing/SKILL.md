# E2E Browser Testing

Проверять поведение так, как его видит пользователь — в реальном браузере, а не только
юнитами. Инструмент по умолчанию — **Playwright** (Chromium предустановлен в среде кита).

## Что проверяем

- **Сценарии, а не загрузку страницы**: пройти реальный путь пользователя (клик, ввод,
  переход), а не только «страница отрендерилась».
- **Состояния**: Empty / Loading / Error / Success — каждое видимо и проверено.
- **Адаптивность по матрице устройств**: тесты гоняются на **viewport-матрице** из
  `context/product/DesignSystem.md` (Playwright projects по классам mobile/tablet/desktop),
  а не только на одном десктопе — это замыкает пункт `e2e-viewport-matrix` из
  `rules/design/responsive-baseline.yaml`.
- **Доступность на пути**: фокус с клавиатуры, роль/имя ключевых элементов.

## Как гоняем

- **Стабильные селекторы**: роль/`data-testid`, не хрупкие CSS-цепочки и не текст, зависящий
  от локали.
- **Скриншоты и консоль при падении** — артефакты для диагностики; визуальные снапшоты
  для регрессий по классам матрицы.
- **CI — headless и детерминированно**: фиксированные размеры вьюпорта, ожидания по
  состоянию (не `sleep`), сеть замокана где нужно.
- Результат — вход для гейта `implementation_verification` (evidence: tests_passed на ревизии).

## Границы

- Не подменяет юнит/контрактные тесты — дополняет их на уровне пользователя.
- Не тестирует сторонние потоки, которые невозможно наблюдать (см. границы в
  `skills/product-session-review`).
- Не хардкодит секреты в тесты (`governance/information-boundaries.md`).

## Отказы

- не «зелёный» тест, который ничего не проверяет (нет ассертов на состояние/результат);
- не хрупкие селекторы и `sleep` вместо ожиданий;
- не гонять только десктоп, если матрица устройств шире.
