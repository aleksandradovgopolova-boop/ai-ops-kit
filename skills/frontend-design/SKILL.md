# Frontend Design

Цель — интерфейс с точкой зрения, а не «подойдёт любому проекту». У кита сильное
*ревью* UI (ux-heuristics, design-system, responsive, accessibility); здесь — дисциплина
*создания*. Работает с обычным HTML/CSS, без привязки к фреймворку.

## Два прохода (план → критика → сборка)

1. **План, укоренённый в предмете.** До кода — дизайн-направление из сути продукта:
   палитра, пары шрифтов, идея макета, «фирменный» элемент. Не абстрактная красота, а
   отражение того, о чём продукт.
2. **Критика против generic.** Прежде чем верстать, честно спросить: это не одно из
   типовых AI-клише? Избегать трёх кластеров: «кремовый + serif», «тёмный + неон»,
   «broadsheet/газета». Один **оправданный** эстетический риск лучше десяти безопасных
   штампов.
3. **Сборка.** HTML/CSS по плану; из токенов дизайн-системы (`context/product/DesignSystem.md`),
   не ad-hoc.

## Границы качества (не украшательство)

- **Копирайт служит пониманию**, а не декорации; текст — часть дизайна.
- **Движение оркестровано**, а не разбросано; уважать `prefers-reduced-motion`.
- **Сложность соответствует замыслу** — не усложнять ради вида.
- **Доступность и адаптивность — не опция**: клавиатурный фокус, контраст, матрица
  устройств (`rules/design/responsive-baseline.yaml`), состояния Empty/Loading/Error/Success.

## Стык с китом

- Источник токенов и матрицы устройств — `context/product/DesignSystem.md` (не выдумывать).
- Результат проходит ревью-гейты `ux_review` и `design_system_usage` — этот скилл готовит
  UI так, чтобы их пройти по существу, а не подгонкой.
- e2e/визуальные регрессии по viewport-матрице ловит скилл `e2e-browser-testing`.

## Отказы

- не выдавать generic-шаблон за дизайн; не игнорировать предметную область;
- не заводить цвета/шрифты/отступы мимо токенов дизайн-системы без обоснования;
- не жертвовать доступностью и адаптивностью ради вида.
