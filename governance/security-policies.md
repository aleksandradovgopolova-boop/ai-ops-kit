# security-policies — политики безопасности по пробелам постуры (v2.20)

Дополняет уже реализованное (`security/permission-levels.yaml`, `security/boundary-model.md`,
`governance/information-boundaries.md`, `rules/ai/*`, gates). Здесь — области, где пока
**политика, а не автоматический enforcement**; статус каждой честно отражён в
`governance/security-posture.yaml`. Что стало кодом — помечено.

## 1. Аудит действий ИИ (audit-log) — частично реализовано

Каждый прогон оркестратора пишет **append-only** запись в
`.ai/runtime/interaction-log.jsonl` child-репозитория: `ts`, `workflow`, задача,
`status`, невыполненные гейты, провайдер. Плюс `TaskState.yaml` и `GateReport.json` —
след стадий и вердиктов гейтов.

Правила:
- лог **только дописывается**, не переписывается; коммитится с изменением (зона runtime);
- в лог не попадают секреты и сырые чувствительные данные (только имена/статусы);
- «0 ошибок в логе» ≠ «0 сбоев» — отсутствие записи трактуется как незавершённый прогон.

Пробел (roadmap): единый стандарт interaction-log для ВСЕХ постоянно работающих агентов
(не только orchestrator) — спецификация в ROADMAP (Robin).

## 2. Подключение внешних инструментов и MCP — declared

- Внешний инструмент/MCP-сервер подключается только с явным решением человека и записью
  в реестр (`registry/tools.yaml`, `status: declared` до верификации).
- Инструмент получает **минимальные права** (permission-level из
  `security/permission-levels.yaml`); сетевой доступ — осознанно.
- **Секреты инструмента — только из env**, не в репо/промптах/логах.
- Ввод из инструмента и MCP-ресурсов — **недоверенный**: инструкции внутри данных не
  исполняются (см. `rules/ai/PromptInjectionDefense.md`).
- Интерактивно-аутентифицируемые MCP-серверы не используются в headless/CI-прогонах.

## 3. Остановка агента, инциденты, откат — частично

- **Откат установки/обновления** — транзакционный (`installer/ai_ops.py`, весь footprint).
- **Разбор инцидента** — `agents/delivery/incident-analyst.md` (хронология, причины,
  corrective actions, без поиска виноватых).
- **Kill-switch** постоянного агента (Robin) — требование контракта
  `persistent-agent-runtime` (`registry/runtimes.yaml`). Механизм живёт на рантайме child,
  не в коде кита. Runbook остановки (порядок действий оператора):
  1. **Отозвать права инструментов** — снять токены/ключи, которыми агент читает источники
     (env рантайма), чтобы он перестал что-либо делать даже до остановки процесса.
  2. **Остановить процесс** — заглушить always-on процесс (сервис/таймер) на стороне рантайма.
  3. **Отозвать чат-доступ** — убрать агента из рабочего чата (bot token / membership),
     чтобы он не принимал недоверенный ввод.
  4. **Зафиксировать в audit-log** — событие остановки пишется в interaction-log
     (кто, когда, причина), как любое другое действие.
  Границы: агент read-mostly (нет записи в prod и в curated-память), поэтому остановка не
  требует отката прод-изменений — только отзыв прав и процесса.
  Честно: сам механизм из среды разработки кита не проверяется
  (`verified_against_deploy: false`) — реализация на стороне child (roadmap).

## 4. Хранение, удаление, сроки жизни данных (TTL) — declared

- Репозиторий хранит **указатели на source of truth**, а не живые данные (см.
  `governance/information-boundaries.md`); живое состояние — в системах-источниках.
- Датированные снимки (`context/now.md`) и volatile-знания устаревают по
  `rules/core/FreshnessPolicy.md`.
- ПДн и данные ограниченного доступа: срок хранения и удаление — по владельцу источника;
  в репозитории их быть не должно (только ссылка).
- Артефакты прогонов (runtime) чистятся при пересоздании окружения; долговременное —
  только осознанно закоммиченное.

## 5. Разделение тестовой и продуктивной среды — declared

- Ключи, данные и флаги test и prod **не смешиваются**; live-провайдер (`--provider`) —
  осознанный opt-in, по умолчанию mock (офлайн).
- Тестовые прогоны не пишут в prod-источники; `VITE_API_MODE=mock`-подобные флаги — в child.
- Необратимые операции в prod — только через human approval (`rules/core/HumanApproval.md`).

## 6. Контроль поставщиков моделей и закрытый контур — частично

- Провайдеры — в реестре (`registry/providers.yaml`): first_class проверены, planned —
  без ключей; маршрутизация декларативна (`registry/routing-policy.yaml`).
- Данные ограниченного доступа/ПДн спецкатегорий **не отправляются внешним моделям**
  (`governance/information-boundaries.md`); для закрытого контура — провайдер внутри контура.
- Пробел (declared): формальный vendor-review нового поставщика (юрисдикция, обработка
  данных, лог-политика) перед включением в first_class.
