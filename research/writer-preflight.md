# Writer pre-flight — чек-лист перед отправкой DP на ревью

Источник: классы major-находок независимых судей по DP-102…DP-106 (два раунда каждый).
Цель: закрывать типовые дефекты ДО ревью — типовой цикл должен стать одним раундом.
Прогоняется writer'ом по каждому пункту; «не применимо» — валидный ответ, «не проверил» — нет.

1. **Каждый пункт rationale опирается на процитированные EV-id.** Нет утверждений
   из головы, оформленных ссылкой на EV, который их не содержит (DP-102 r1: dev-browser/agnix).
2. **Каждый constraint запроса адресован явно** — соблюдён, ослаблен с обоснованием
   или признан unknown (DP-102 r1: no_new_python_deps; DP-105 r1: additive_only).
3. **Негативные утверждения** («X отсутствует / не найдено») — с квалификаторами,
   охватом проверки и собственным EV; reliability не выше medium (DP-103, DP-105 r1).
4. **Оговорки EV не теряются при переносе в рекомендацию.** Если EV фиксирует
   ограничение (privacy, лицензия, scope) — оно едет в rationale/декларацию, не остаётся
   в evidence (DP-106 r1: hf-router без privacy-scope из EV-402).
5. **Цифры не переносятся на другой дизайн.** Метрика измерена для конфигурации A —
   нельзя оправдывать ей конфигурацию B (DP-104 r1: дебаты → одиночный судья).
6. **cost_of_recommendation заполнен**: разовое, постоянное, владелец. Стоимость
   эксплуатации — не только аренда/лицензии (DP-106 r1: TCO стека).
7. **Пороги proposed_validation фальсифицируемы**: числа, а не «сопоставимо/работает»;
   критерий провала определён; конфаундеры разделены в учёте (DP-104/105/106 r1).
8. **Валидация покрывает рекомендацию**, а не 10% её; невалидируемое — явно помечено
   «отложено» (DP-104 r1). Пилот с малым n отсеивает провал, но не доказывает превосходство.
9. **Сверка с существующими артефактами кита/продукта**: registry, policies, rules —
   рекомендация не создаёт вторую политику и не противоречит действующей
   (DP-105 r1: EvalTooling.md; DP-106 r1: routing-policy).
10. **Внутренняя консистентность полей**: preferred_option ↔ rationale ↔ assumptions ↔
    risks ↔ validation не противоречат друг другу; числа/названия совпадают
    (DP-104 r1: optional vs required; DP-106 r2: Team/Enterprise, 32B/33B).
11. **Обе стороны сравнения эвидентны.** Позиционирование/сравнение с чужим продуктом
    требует EV и о своём (DP-103 r1: кит без evidence).
12. **decision_brief написан** — 5-7 строк для решателя: рекомендация / цена /
    топ-риски / следующий шаг.
13. **Гигиена evidence**: один источник — одна запись; quote соответствует своему
    source.url; inference имеет derived_from; self-evidence помечен для спот-чека judge.
14. **Review-блок пуст** (заполняет координатор из вердикта judge, не writer).
15. **Продуктовая потребность ≠ наша capability.** Дифференциация не выводится из того,
    что мы умеем (tech-push); потребность пользователя либо эвидентна, либо помечена
    гипотезой с проверкой в validation (DP-107 r1: verifiability-цепочка).
16. **Пороги согласованы с n и с ролью.** Число респондентов/проб зафиксировано; малый n
    даёт только стоп-сигнал, формула «подтверждается при прохождении» с малым n запрещена;
    протокол отбора респондентов задан (DP-107 r1: «3-5 интервью» vs порог «из 5»).

Чек-лист живой: новые классы находок судей добавляются сюда же (контур калибровки writer'а).
