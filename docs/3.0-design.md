# AI Ops Kit 3.0 — дизайн: `ai-ops run` основным путём + сплит на 5 пакетов

Статус: **план (design)**, не реализация. Запускается по явному решению владельца. Это
детальный план эпика — как v2.31 запускали фазами, так и 3.0 идёт срезами с валидацией
каждого.

## Зачем 3.0

К версии 2.44 кит вырос из «конституции» (registry → workflow → agents → gates) в исполняющее
ядро (RunPlan, Tool Broker, budget, tool-loop, evidence collector). Два накопленных долга —
**breaking по своей природе**, поэтому вынесены в major:

1. **Единый вход.** Сейчас основной путь — `/ai-start-task` (runtime-команда), а `ai-ops run`
   (единый контроллер: route → RunPlan → WorkItem → active-work → исполнение → отчёт) —
   параллельный. Разные пути расходятся; хочется один канонический.
2. **Монолит пакета.** Всё ставится одним `installer/ai_ops.py init`. Ребёнку, которому нужен
   только quality-слой или только execution, приходится тянуть весь кит. Гранулярность
   невозможна без разделения на пакеты.

## Что breaking (и почему это 3.0, а не 2.x)

- Канонизация `ai-ops run` меняет **дефолтный** контракт запуска (старый путь становится
  совместимым алиасом, но «основной путь» — другой).
- Сплит меняет **структуру установки** (`.ai/managed/**`), пути пакетов, формат `.ai-ops.yaml`
  (`parent.source` → набор пакетов с версиями). Это ломает предположения существующих child.

Инвариант совместимости 2.x (аддитивность, без переносов файлов) специально снимается — но
только под защитой миграционного моста (ниже).

## Границы пяти пакетов

Разрез — по **зонам ответственности** (карта в AGENTS.md), с минимумом связей между пакетами.

| Пакет | Содержимое (из текущего дерева) | Роль |
|---|---|---|
| **ai-ops-core** | `registry/`, `manifest/`, `schemas/`, `config/`, валидаторы целостности реестра (`validate_ai_first_*`, `ai_route`, `ai_capability_selftest`) | Конституция: источники истины и маршрутизация. От него зависят все. |
| **ai-ops-quality** | `quality/gates.yaml`, `tools/gate_executor.py`, `validate_reviewer_result`, `tools/project_detector.py`, `tools/evidence_collector.py`, `rules/quality`, `rules/design`, `rules/engineering` (review) | Гейты, evidence, ревью. Зависит от core. |
| **ai-ops-execution** | `tools/orchestrator.py`, `tool_broker`, `budget`, `tool_loop`, `run_plan`, `ai_ops_run`, `active_work`, `worktree`, `concurrency_preflight`, `merge_memory`, `registry/tracks.yaml`, `registry/runtimes.yaml` | Исполняющее ядро. Зависит от core + quality. |
| **ai-ops-product** | `agents/`, `evaluations/`, `templates/`, `skills/`, blueprint-инструменты, `product_health`, knowledge-graph, `effect_metrics`, workflows PRODUCT/INSIGHTS/ADOPTION/AI_FEATURE, `context/`, `memory/` | Продуктовый слой: агенты, артефакты, метрики эффекта. Зависит от core. |
| **ai-ops-installer** | `installer/ai_ops.py`, `templates/ci/*`, онбординг (`ONBOARDING.md`), update/rollback flow, `.ai-ops.yaml`-контракт | Композитор: ставит выбранные пакеты в child, версионирует, обновляет. Зависит от всех (оркеструет установку). |

Правило зависимостей — **однонаправленное**: `core ← quality ← execution`, `core ← product`,
`installer → {все}`. Циклов нет; валидатор границ (новый) это проверяет.

## `ai-ops run` основным путём

- `ai-ops run "<задача>"` — канонический вход: классификация → RunPlan → WorkItem →
  preflight/active-work → исполнение (claude-code: тонкий контроллер + evidence; generic:
  tool-loop) → run-report.
- `/ai-start-task` остаётся **совместимым алиасом** (делегирует в `ai-ops run`), помечается
  deprecated-но-поддерживается; удаление — не раньше 4.0.
- README/QUICKSTART/commands переописываются от `ai-ops run` как основного (в 2.45 —
  аддитивная промо-часть без снятия старого; полная канонизация — в 3.0).

## Миграционный мост (child не должен «сломаться в один день»)

1. **Монорепо-пакеты сначала.** 3.0.0 держит все 5 пакетов в ЭТОМ репозитории с отдельными
   под-манифестами (`packages/<name>/package.yaml`); отдельная дистрибуция — позже (3.1+).
2. **Мета-пакет по умолчанию.** `installer init/update` по умолчанию ставит все 5 (как сейчас
   — единый footprint). Существующий `ai-ops update` продолжает работать; child не замечает
   разницы, пока не выберет гранулярность.
3. **`.ai-ops.yaml` — с дефолтом.** Новое поле `packages:` (список с версиями) опционально;
   отсутствует → «все пакеты» (обратная совместимость чтения).
4. **Транзакционность сохраняется.** Обновление остаётся rollback-safe (smoke-валидаторы →
   откат всего footprint), теперь — по-пакетно с общим backup.
5. **MIGRATION_GUIDE 2.x → 3.0** с точными шагами и «что стало по-другому».

## Срезы реализации (каждый — свой валидатор + selftest, как в v2.31–2.44)

- **3.0-срез 0:** валидатор границ пакетов (dependency-graph без циклов) + карта
  файл→пакет (`packages/*/package.yaml`), БЕЗ переноса файлов (только декларация). Аддитивно,
  можно выпустить в 2.x как подготовку.
- **3.0-срез 1:** `ai-ops run` канонический; `/ai-start-task` → алиас; docs переописаны.
- **3.0-срез 2:** installer ставит по-пакетно (мета-дефолт = все 5); `.ai-ops.yaml.packages`.
- **3.0-срез 3:** физический разнос дерева по `packages/<name>/` + обновление всех путей
  и CI; обкатка на ии-среде через `ai-ops update`.
- **3.0-срез 4:** MIGRATION_GUIDE + релиз 3.0.0; период поддержки 2.x-моста.

## Риски и почему нужен явный запуск

- Разнос дерева трогает пути в десятках валидаторов/CI/командах — высокий blast radius.
- Меняется установочный контракт всех child (включая ии-среду) — outward-facing, тяжело
  откатывается после того, как child мигрировали.
- Поэтому 3.0 стартует **только по явному «да»** и идёт срезами с обкаткой на реальном child,
  а не одним коммитом.

## Не входит в 3.0 (осознанно)

- Постоянный runtime кита / Robin как бот (остаётся runtime-binding у child).
- Отдельная публикация пакетов в реестр (pip/npm) — это 3.1+, после стабилизации границ.
