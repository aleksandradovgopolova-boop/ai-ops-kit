# Параллельные сессии: несколько задач в одном репозитории

Можно открывать несколько сессий Claude на один репозиторий и вести разные задачи
одновременно — это ускоряет работу. Конфликтов не будет при одном правиле:

> **Одна сессия = своя ветка = своя область = свой PR.**

## Почему это безопасно

Каждая веб-сессия Claude работает в **своём изолированном окружении со свежим клоном**
репозитория. Сессии **не делят рабочую папку** — одна не затирает файлы другой на диске.
Встречаются они только в git на GitHub (ветки и слияния), поэтому единственный возможный
конфликт — обычный git-конфликт при мердже, а не хаос в реальном времени.

## Правила (чтобы точно без боли)

1. **Каждой сессии — своя ветка** (`feature/catalog`, `feature/admin`) и свой PR.
   Не давайте двум сессиям писать в одну ветку.
2. **Делите работу по непересекающимся файлам/фичам.** Сессия 1 — экран каталога,
   сессия 2 — админка. Тогда и мерджить нечего разруливать.
3. **Мерджите по одной.** Влили A → в ветке B сделайте rebase на свежий main.
4. **Не запускайте `ai-ops update` в двух сессиях сразу** — обновление кита ведите в
   одной ветке.

## Где конфликт может «спрятаться»

Опаснее не текстовые git-конфликты (их git видит), а **смысловые** — когда две сессии
правят общий источник истины, git слил чисто, а логика разошлась:

- `registry/*`, `context/product/DesignSystem.md`, `context/product/MetricCatalog.md`,
  `analytics/tracking-plan.md` одной фичи, `decisions/registry.yaml`.

> Правило: **общий source-of-truth правит только одна сессия за раз.**

## Чем страхует кит

Кит не предотвращает коллизии магически, но ловит рассинхрон **на CI/PR**, пока не поздно:

- `validate_references` — битые ссылки после параллельных правок;
- `validate_cross_artifacts` — tracking-plan и дашборд разошлись;
- claims/drift и стейл-гейты — документация против кода;
- writer ≠ judge + независимое ревью на каждом PR.

## Честная граница

Сам кит **не запускает параллельных агентов** — оркестратор sequential-only. Параллелизм
создаёте вы, открывая несколько сессий; кит даёт ветки-контракты и проверки как страховку
при слиянии. Столько сессий, сколько задач, — просто по ветке на каждую и мердж по очереди.
