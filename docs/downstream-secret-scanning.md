# Downstream secret-scanning: как не блокировать обновления AI Ops Kit

Кит — это **security-инструмент**, поэтому его исходники и документация закономерно содержат лексику
безопасности: слова `secret`, `token`, имена переменных окружения (`GITHUB_TOKEN`, `OPENAI_API_KEY`,
`ANTHROPIC_API_KEY`, …), домен `secrets`, модуль `security_scan`. Это **имена и документация, а не
значения секретов.**

Секрет-сканеры бывают двух классов:

- **value/entropy/format-based** (правильный): флагует реальные значения — `AKIA[0-9A-Z]{16}`,
  `sk-[A-Za-z0-9]{20,}`, приватные ключи (PEM-блоки), высокоэнтропийные строки. На лексику не реагирует.
- **keyword-based** (слишком широкий): флагует по словам `secret`/`token` или по именам переменных.
  Такой даёт **100% ложных срабатываний** на любой security-документации — и блокирует PR обновления
  кита, хотя реальных секретов нет.

Если PR обновления кита блокируется на строках вроде `GITHUB_TOKEN` / `secret` в документации — это
**правило сканера слишком широкое**, а не утечка. Ниже — санкционированные способы это исправить
(не обходя сканер: реальные секреты в **вашем** коде по-прежнему сканируются).

## Гарантия пакета

В ките **нет реальных секретов и нет статических секрет-подобных литералов** (проверяется на каждом
релизе; фикстуры детектора собираются в рантайме из фрагментов, см. CHANGELOG v3.0.4). Поэтому
исключить вендоренный managed-слой из секрет-скана — **безопасно**.

## Способ 1 (рекомендуемый): исключить вендоренный managed-слой

`.ai/managed/**` — это **вендоренная зависимость** (файлы кита, скопированные в ваш репо), как
`node_modules/` или `vendor/`. Секрет-сканер вашего репозитория должен сканировать **ваш** код, а не
документацию сторонних пакетов. Исключите путь managed-слоя:

**gitleaks** (`.gitleaks.toml` в корне репо):

```toml
[allowlist]
description = "AI Ops Kit — вендоренный managed-слой (security-инструмент, только лексика, не значения)"
paths = ['''\.ai/managed/.*''']
```

**trufflehog** — файл исключений (`--exclude-paths trufflehog-exclude.txt`):

```
.ai/managed/
```

**detect-secrets** (`--exclude-files`, либо в baseline):

```
^\.ai/managed/.*
```

**GitHub secret scanning / gitleaks-action** — добавьте `.ai/managed/**` в `paths-ignore`/allowlist
конфигурации экшена.

Реальные секреты в вашем собственном коде (`src/`, конфиги и т.п.) продолжают сканироваться.

## Способ 2: сузить правило до value/format-based

Если вы не хотите исключать путь, замените keyword-правило на детекцию **значений**:

- AWS access key: `AKIA[0-9A-Z]{16}`
- OpenAI-подобные ключи: `sk-[A-Za-z0-9]{20,}`, `sk-ant-[A-Za-z0-9_-]{20,}`
- GitHub-токены: `gh[pousr]_[A-Za-z0-9]{20,}`, `github_pat_[A-Za-z0-9_]{20,}`
- Приватные ключи: PEM-заголовок блока приватного ключа
- Высокая энтропия присвоенных строковых литералов

Имена переменных (`GITHUB_TOKEN`, `OPENAI_API_KEY`) и слова (`secret`, `token`) в правило **не входят** —
они не являются секретами.

## Почему не «просто убрать слова из пакета»

Убрать `secret`/`token`/имена ключей из инструмента, который **сам** про секреты, security-скан и
доставку через `GITHUB_TOKEN`, — невозможно и неправильно: это сломает документацию и смысл. Правильная
граница: сканер отвечает за **значения секретов**, а не за лексику; вендоренные зависимости из скана
исключаются штатно.
